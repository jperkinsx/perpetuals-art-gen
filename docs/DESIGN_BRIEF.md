# PERPETUALS — Generative Art Design Brief

## For: LLM-assisted generative art development
## Project: perpetuals.art
## Author: Jonathan Perkins / SuperRare

---

## 1. Project Context

**Perpetuals** is a self-perpetuating autonomous art project built on [Rare Protocol](https://rare.xyz) and [Bankr](https://bankr.fi). Every 24 hours, an AI agent mints a new piece of generative art, auctions it via SuperRareBazaar, and reinvests proceeds into a treasury — forever. There is no human operator. The system runs autonomously: creating, selling, and funding itself in a closed loop.

The art is the heartbeat of this system. Each piece represents one "epoch" — a single day in an infinite cycle. The collection grows by one every day, permanently.

### Architecture (for context)
- **Perpetual.sol** — ERC-721 contract, mints one token per epoch
- **SuperRareBazaar** — 24-hour timed auctions with anti-snipe extensions
- **BankerVault** — Treasury that earns yield and funds operations
- **Sentinel** — AI agent that monitors social signals and scores engagement authenticity (anti-sycophancy)

### What makes this different from other generative projects
- **Infinite collection**: No cap. One new piece per day, forever.
- **Autonomous**: No artist hand. The algorithm is the artist.
- **Economic feedback loop**: Auction revenue feeds the treasury, which funds the next mint. If the art stops selling, the system adapts.
- **Sycophancy-resistant**: The Sentinel agent evaluates whether community engagement is genuine or bot-driven spam. The art system should reflect this ethos — rewarding signal over noise.

## 2. Hard Constraints

### Must be fully on-chain
The art output **must** be storable directly in the smart contract or derivable entirely from on-chain data (tokenId, block hash, timestamp, epoch number, treasury balance, etc.). No IPFS. No Arweave. No external dependencies.

### Lightweight
The rendering output should be one of:
- **ASCII / Unicode text art** (like Autoglyphs — raw characters in a grid)
- **SVG markup** generated by the contract (like on-chain SVG NFTs)
- **HTML + minimal JS** stored as a base64 data URI in tokenURI

Target: **under 10KB per token** for the rendered output. Ideally much less. The generation logic lives in the contract; the per-token cost should be near-zero marginal storage.

### Deterministic
Given the same seed inputs, the art must always produce the same output. Seeds should derive from on-chain data:
- `tokenId` (epoch number: 1, 2, 3, ...)
- `block.timestamp` at mint time
- `block.prevrandao` or `blockhash`
- Optionally: treasury balance at mint time, auction settle price of previous epoch

## 3. Aesthetic Direction

### Mood
- **Eternal machine**: This art is made by a system that will outlive its creators. It should feel like watching a clock that never stops — meditative, rhythmic, but never repetitive.
- **Quiet intensity**: Not loud or flashy. The beauty should reveal itself on close inspection. Think: the satisfying patterns in static, the hypnotic quality of a loading terminal, the elegance of a mathematical proof.
- **Digital organism**: Each epoch is a heartbeat. The collection, viewed in sequence, should feel like watching something alive — breathing, shifting, evolving.

### Visual Language
- **Monochrome + one accent color**: Black background. White or light gray primary. Single accent color: **phosphor green** (`#22c55e` / `hsl(145, 63%, 49%)`). This is the project's signature color — terminal green, the color of a cursor blinking in the dark.
- **Grid-based composition**: Art should be composed on a grid (e.g., 32×32, 48×48, 64×64 characters or cells). Grids evoke screens, terminals, spreadsheets, ledgers — all relevant metaphors.
- **Typographic / symbolic**: Use ASCII characters, Unicode box-drawing characters, mathematical symbols, or Braille patterns as building blocks. The "pixels" of this art are characters, not colors.

### Character Palette (suggested, not prescriptive)
```
Structural:  ┌ ┐ └ ┘ ─ │ ┬ ┴ ├ ┤ ┼ ╔ ╗ ╚ ╝ ═ ║
Density:     ░ ▒ ▓ █ ▀ ▄ ▌ ▐
Geometric:   ◆ ◇ ◈ ○ ● ◐ ◑ ◒ ◓ △ ▽ ▷ ◁ ⬡ ⬢
Symbolic:    ∞ ∅ ∆ ∇ ∴ ∵ ≈ ≡ ⊕ ⊗ ⊞ ⊠
Dot/Braille: ⠀ ⠁ ⠂ ⠃ ⠄ ⠠ ⠰ ⣀ ⣇ ⣿
Arrows:      → ← ↑ ↓ ↗ ↘ ↙ ↖ ⟳ ⟲
Currency:    Ξ ◊ (subtle nods to ETH / crypto)
```

### Recognizability
The art must be **instantly identifiable** as a Perpetuals piece. Like how you can spot a Nouns glasses or an Autoglyph from a thumbnail. This means:

1. **Consistent structural motif**: Every piece should share a recognizable compositional element — a frame, a central form, a recurring symbol. Suggestion: a **circle or ring** as the recurring anchor (representing the perpetual cycle). The circle can be rendered in ASCII/Unicode and should appear in every piece, but its internal content varies per epoch.

2. **Epoch number as part of the art**: The epoch number (001, 002, 003...) should be visually embedded in the piece — not as a label, but as part of the composition. Think of it like a date stamp etched into the frame.

3. **Consistent dimensions**: Every piece should be the same grid size. This creates uniformity when displayed as a collection/gallery.

## 4. Generative Parameters

The art should vary across epochs using seed-derived parameters. Suggested axes of variation:

| Parameter | Range | Effect |
|-----------|-------|--------|
| **Density** | Sparse → Dense | How filled the grid is. Early epochs might be sparse; density could correlate with treasury health. |
| **Symmetry** | None / Bilateral / Radial / 4-fold | Type of symmetry applied to the pattern. |
| **Symbol set** | 4–6 curated sets | Which characters are used. Different sets create different textures. |
| **Ring radius** | Small → Large | Size of the central circular motif. |
| **Internal pattern** | Waves / Spirals / Noise / Lattice / Void | What fills the interior of the ring. |
| **Border style** | Thin / Double / Dotted / None | Frame around the piece. |
| **Accent density** | 0–15% of cells | How many cells use the green accent vs. white/gray. |

### Seed derivation (pseudocode)
```
seed = keccak256(abi.encodePacked(tokenId, block.timestamp, block.prevrandao))
density    = uint8(seed[0]) % 5      // 0-4
symmetry   = uint8(seed[1]) % 4      // none, bilateral, radial, quad
symbolSet  = uint8(seed[2]) % 6      // which character palette
ringRadius = uint8(seed[3]) % 8 + 4  // 4-11 cells
pattern    = uint8(seed[4]) % 5      // wave, spiral, noise, lattice, void
border     = uint8(seed[5]) % 4      // thin, double, dotted, none
accentPct  = uint8(seed[6]) % 16     // 0-15% green cells
```

## 5. Reference Projects

### Autoglyphs (Larva Labs, 2019)
- 512 pieces, fully on-chain, ASCII character art on a 64×64 grid
- 10 symbol sets, patterns generated from a simple algorithm in Solidity
- **What to take**: Simplicity, recognizability, the power of constrained character palettes
- **What to improve**: More visual depth, color (even just one accent), stronger compositional identity

### Terraforms (Mathcastles, 2021)
- 9,911 pieces, animated Unicode art on a 32×32 grid, fully on-chain
- Custom font, color palettes ("Zones"), animated character cycling
- **What to take**: The richness achievable with Unicode, the idea of zones/biomes creating variation families
- **What to improve**: Perpetuals should be more immediately readable at thumbnail size — Terraforms can be abstract to the point of illegibility

### Loot (Dom Hofmann, 2021)
- Text-only NFTs (white text on black background), fully on-chain SVG
- Randomized word lists rendered as simple typography
- **What to take**: The radical simplicity, the power of text-as-art, the composability
- **What to improve**: Perpetuals should have visual richness beyond plain text — the characters themselves should form images

### Chromie Squiggle (Snowfro / Art Blocks, 2020)
- Simple algorithm, one visual motif (a squiggle), massive variation within constraint
- **What to take**: The "one motif, infinite variation" approach — this is the gold standard for recognizability

## 6. Output Format

### Preferred: SVG with embedded monospace text
```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 640 640">
  <rect width="640" height="640" fill="#0a0a0a"/>
  <text font-family="monospace" font-size="10" fill="#e5e5e5">
    <tspan x="10" y="20">┌──────────────────────────────────┐</tspan>
    <tspan x="10" y="30">│  ░░░░░▒▒▒▓▓▓███◆███▓▓▓▒▒▒░░░░░ │</tspan>
    <tspan x="10" y="40">│  ░░░▒▒▓▓██  ╭───────╮  ██▓▓▒▒░░░ │</tspan>
    <!-- ... rows of the grid ... -->
    <tspan x="10" y="330">│              EPOCH 001            │</tspan>
    <tspan x="10" y="340">└──────────────────────────────────┘</tspan>
  </text>
  <!-- Optional: green accent overlay for select characters -->
  <text font-family="monospace" font-size="10" fill="#22c55e">
    <tspan x="210" y="170">◆</tspan>
    <!-- accent characters at specific positions -->
  </text>
</svg>
```

### Alternative: Pure ASCII string (Autoglyphs-style)
Return the art as a raw string stored in the contract. The frontend renders it in a `<pre>` tag with the project's monospace font and color scheme. Even cheaper on gas — the contract just stores/generates a string.

### Rendering in tokenURI
The contract's `tokenURI()` function should return a base64-encoded JSON with the SVG (or ASCII) embedded:
```json
{
  "name": "Perpetuals #001",
  "description": "Epoch 001. A self-perpetuating artwork.",
  "image": "data:image/svg+xml;base64,PHN2Zy..."
}
```

## 7. What We're Asking the LLM to Help With

1. **Design the core algorithm**: Given a seed (uint256), produce a grid of characters that forms a visually compelling, recognizable piece of art following the aesthetic direction above. The algorithm should be expressible in Solidity or in a minimal JS renderer (< 2KB).

2. **Design 3–5 "trait families"**: Axes of variation (like Autoglyphs' symbol sets or Terraforms' biomes) that create distinct visual categories while maintaining the unified Perpetuals identity.

3. **Produce 10 example outputs**: Show what epochs 001 through 010 would look like as raw ASCII/Unicode text grids. These should demonstrate the range of variation.

4. **Define the visual signature**: What is the single most recognizable element? (Proposed: the ring/circle motif.) Finalize this and show how it appears across different parameter combinations.

5. **Optimize for beauty at multiple scales**: The art should look good as:
   - A 64×64 full-resolution view (detail)
   - A ~200px thumbnail in a gallery grid (recognizability)
   - A Twitter/Farcaster profile picture (icon legibility)
